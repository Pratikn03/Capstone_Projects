\section{ORIUS Runtime Architecture}
\label{sec:prof-runtime-architecture}

ORIUS is a post-nominal runtime layer. It does not replace the upstream predictor,
planner, optimizer, or policy. It wraps them with a narrow contract that begins at raw
telemetry and ends at certificate-gated release. This is the architectural reason the
framework can remain domain-agnostic even when the domains do not share nominal models
or action geometry.

The kernel consists of five stages.
\begin{enumerate}[leftmargin=1.25em,itemsep=0.2em]
  \item \textbf{Detect:} estimate reliability \(w_t\) and drift flags from freshness,
  missingness, cross-sensor consistency, residual behavior, and domain-specific
  diagnostics.
  \item \textbf{Calibrate:} inflate the observation-consistent set \(U_t\) rather than
  trusting a nominal point estimate.
  \item \textbf{Constrain:} recompute the safe action set under the widened state set.
  \item \textbf{Shield:} keep, repair, or replace the candidate action.
  \item \textbf{Certify:} emit a causal certificate and update the governance state.
\end{enumerate}

\subsection{Execution path through the kernel}
The practical execution path is simple enough to be implemented in a few typed calls,
but rich enough to support formal analysis. A telemetry packet enters through the
domain adapter with time stamps, missingness indicators, residual channels, and any
domain-specific diagnostics. The detect stage converts those observations into a
reliability score and degradation flags. The calibrate stage then refuses the usual
engineering shortcut of keeping a point state estimate when trust has fallen; instead
it materializes the observation-consistent state set that the controller must now
respect. The constrain stage computes what actuation remains safe for every state in
that set. Shielding then either preserves the candidate action, repairs it into the
safe set, or invokes an explicit fallback. Certify binds the release decision to a
certificate payload and governance state transition.

That same execution path has different physical interpretations across domains. In
battery systems it becomes charge or discharge limitation under hidden state-of-charge
uncertainty. In AV it becomes governed braking or maneuver restriction under degraded
headway confidence. In healthcare it becomes intervention legality under uncertain
vital-sign trust. The point of ORIUS is not that these domains share dynamics. The
point is that they share a release-boundary problem once degraded observation is made
explicit.

The kernel becomes scientific rather than architectural once its objects are written
explicitly. ORIUS uses a reliability-conditioned uncertainty envelope
\begin{equation}
  U_t
  =
  \left\{
    x :
    m_t(x,\hat{x}_t) \le q_{1-\alpha_t}(w_t,\Delta_t)
  \right\},
\end{equation}
where \(m_t\) is a domain-specific mismatch measure and \(q_{1-\alpha_t}\) is a
reliability-aware quantile or inflation operator. The tightened action set is
\begin{equation}
  \mathcal{A}^{\mathrm{safe}}_t(U_t)
  =
  \{a \in \mathcal{A}: g(x,a)\le 0,\ \forall x \in U_t\},
\end{equation}
and the certificate-relevant margin is
\begin{equation}
  \gamma_t(a)
  =
  \inf_{x \in U_t} \phi\!\bigl(f(x,a,0)\bigr),
\end{equation}
so that \(\gamma_t(a) \ge 0\) indicates one-step safety for every state in the
observation-consistent set.

The three equations above are the core of the framework. The first says that trust in
observation is itself a runtime variable. The second says that admissible actuation is
defined against a set, not only against a point estimate. The third says that the
certificate is not a paperwork artifact; it is tied to a concrete safety margin over
that set. Together they turn degraded observation from a monitoring nuisance into a
first-class control-theoretic and systems object.

\begin{algorithm}[t]
\caption{Detect--Calibrate--Constrain--Shield--Certify Runtime Step}
\label{alg:prof-runtime-step}
\begin{algorithmic}
\Require observation packet \(o_t\), history \(\mathcal{H}_t\), candidate action
\(a_t^{\mathrm{cand}}\)
\State compute reliability score \(w_t\) and degradation flags from \(o_t\)
\State build observation-consistent set \(U_t\) using reliability-aware inflation
\State construct tightened safe-action set \(\mathcal{A}^{\mathrm{safe}}_t(U_t)\)
\If{\(a_t^{\mathrm{cand}} \in \mathcal{A}^{\mathrm{safe}}_t(U_t)\)}
  \State \(a_t^{\mathrm{safe}} \gets a_t^{\mathrm{cand}}\)
\ElsIf{\(\mathcal{A}^{\mathrm{safe}}_t(U_t) \neq \varnothing\)}
  \State \(a_t^{\mathrm{safe}} \gets \mathcal{R}_t(a_t^{\mathrm{cand}}, \mathcal{A}^{\mathrm{safe}}_t(U_t))\)
\Else
  \State \(a_t^{\mathrm{safe}} \gets \mathcal{F}_t(c_t)\)
\EndIf
\State emit certificate \(c_t\) with \((w_t,U_t,\gamma_t,\texttt{mode},\texttt{audit\_hash})\)
\State release \(a_t^{\mathrm{safe}}\) only if certificate is valid or fallback is explicit
\end{algorithmic}
\end{algorithm}

This algorithm is the real reusable object in ORIUS. It explains why the framework can
support batteries, vehicles, healthcare monitoring, and future additional domains
without pretending those settings share the same nominal control law. What is shared is
the runtime grammar that mediates degraded observation before actuation.

Two design choices keep this object scientifically clean. First, ORIUS is intentionally
narrow: it sits between nominal intelligence and release rather than trying to subsume
forecasting, planning, or identification. Second, the runtime layer is typed all the
way through. Reliability, uncertainty, repair mode, fallback mode, certificate state,
and audit continuity are explicit objects. That is why the framework can be benchmarked
and governed as a reusable safety layer instead of described only as a software pattern.

\subsection{Why the architecture is universal}
ORIUS achieves universality by freezing the meaning of a few objects and refusing to
freeze everything else. The meaning of reliability, uncertainty widening, tightened
action legality, repaired release, fallback, and certificate validity is invariant
across domains. Telemetry parsers, nominal controllers, local geometry, and domain
constraints are not invariant and are deliberately delegated to adapters. This split is
what lets the same runtime layer sit above an optimizer in energy systems, a planner in
autonomy, a control stack in process industries, or a monitored intervention pipeline
in healthcare without becoming mathematically empty.
