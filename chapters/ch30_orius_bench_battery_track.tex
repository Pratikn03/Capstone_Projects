
\chapter{ORIUS-Bench: Battery Track}
\label{ch:orius-bench-battery}

This chapter describes the public benchmark release of the battery track
within the ORIUS evaluation framework, enabling reproducible comparison
of observation-aware safety controllers on standardised fault scenarios.

\section{Benchmark Scope and Design Goals}

ORIUS-Bench (Battery Track) is an open, deterministic evaluation harness
designed to answer a single question: \emph{does a battery dispatch
controller remain safe when its telemetry is degraded?}

Design goals:
\begin{enumerate}
  \item \textbf{Reproducibility}: deterministic episode generation from a
    (scenario, seed) pair, eliminating run-to-run variance.
  \item \textbf{Fault isolation}: each fault dimension (dropout, delay,
    spike, stale sensor, drift) is independently controllable.
  \item \textbf{Truth separation}: the harness maintains parallel truth
    and observed trajectories, enabling unambiguous violation attribution.
  \item \textbf{Multi-controller comparison}: any controller conforming
    to the \texttt{step(load\_window, renew\_window, \ldots)} interface
    can be plugged in.
\end{enumerate}

\section{Episode Generation}

Episodes are generated by the \texttt{generate\_episode()} function in
\texttt{src/orius/cpsbench\_iot/scenarios.py}.  Each scenario applies
a specific combination of faults to a base load/renewables/price signal
generated from a seeded NumPy RNG:

\begin{itemize}
  \item \textbf{nominal}: clean telemetry (baseline).
  \item \textbf{dropout}: random packet loss at configurable rate.
  \item \textbf{delay\_jitter}: arrival-time perturbation.
  \item \textbf{out\_of\_order}: reordered telemetry packets.
  \item \textbf{spikes}: multiplicative sensor spikes.
  \item \textbf{stale\_sensor}: frozen readings over multi-step windows.
  \item \textbf{drift\_combo}: simultaneous application of all faults
    plus covariate/label drift after a change-point.
\end{itemize}

\section{Metrics Suite}

The benchmark reports three families of metrics, implemented in
\texttt{src/orius/cpsbench\_iot/metrics.py}:

\begin{description}
  \item[Forecast quality] MAE, RMSE, PICP@90/95, Winkler score, pinball loss.
  \item[Control safety] True-SOC violation rate (TSVR), violation severity
    (mean and P95), intervention rate, ramp-violation rate, BMS trip rate.
  \item[Certificate completeness] Presence rate, missing-field count.
\end{description}

\section{Controller Interface}

A benchmark-compliant controller must expose a \texttt{step()} method
returning at minimum:
\begin{itemize}
  \item \texttt{proposed\_charge\_mw}, \texttt{proposed\_discharge\_mw}
  \item \texttt{safe\_charge\_mw}, \texttt{safe\_discharge\_mw}
  \item \texttt{interval\_lower}, \texttt{interval\_upper}
  \item \texttt{certificate} (optional but recommended)
\end{itemize}

\section{Baseline Controllers}

The battery track ships with eight controllers:

\begin{enumerate}
  \item \texttt{deterministic\_lp}: LP dispatch on point forecast.
  \item \texttt{robust\_fixed\_interval}: worst-case robust LP on fixed intervals.
  \item \texttt{cvar\_interval}: CVaR-optimised scenario dispatch.
  \item \texttt{dc3s\_wrapped}: full DC3S with linear inflation law.
  \item \texttt{dc3s\_ftit}: DC3S with FTIT robust-observation law.
  \item \texttt{aci\_conformal}: adaptive conformal inference baseline.
  \item \texttt{scenario\_robust}: scenario-based robust optimisation.
  \item \texttt{scenario\_mpc}: receding-horizon model predictive control.
\end{enumerate}

\section{Summary}

ORIUS-Bench (Battery Track) provides a reproducible, fault-aware
evaluation framework for battery dispatch controllers under degraded
observation.  By maintaining separate truth and observed trajectories
and sweeping fault severity independently, it enables rigorous safety
comparisons that are impossible with standard forecasting benchmarks.
