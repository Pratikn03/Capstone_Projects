
\chapter{Universal ORIUS: Six-Domain Validation}
\label{ch:battery-to-orius}

This chapter presents the ORIUS universal validation: the DC3S shield is
implemented as one shared degraded-observation repair-and-certify kernel
instantiated across six cyber-physical domains---energy management,
autonomous vehicles, industrial process control, medical monitoring,
aerospace, and navigation.  The claim boundary is tiered rather than
uniform: battery is the reference witness, industrial and healthcare are
defended proof-validated peers, autonomous vehicles remains proof-candidate,
navigation is shadow-synthetic, and aerospace remains experimental.  The
shared result is therefore not ``six domains with equivalent guarantees,'' but
one typed runtime contract whose strongest theorem-to-artifact lineage
remains anchored to the battery reference surface.

\section{Why the ORIUS Result Is Structurally General}

The DC3S pipeline never inspects the internal dynamics of the battery
plant directly.  The canonical universal kernel now consumes six typed
objects:

\begin{enumerate}
  \item An \emph{observation packet} carrying raw telemetry and its parsed
    state view at time $t$.
  \item A scalar \emph{reliability assessment} $w_t \in [w_{\min}, 1]$
    summarising telemetry quality and detector output.
  \item An \emph{observation-consistent state set} $U_t$ over the true state,
    widened conservatively when observation quality degrades.
  \item A domain \emph{safety specification} plus a tightened safe-action set
    $\mathcal{A}_t$ derived from $U_t$.
  \item A \emph{repair decision} $a_t^{\text{safe}}$ that maps any candidate
    action into $\mathcal{A}_t$ under the worst-case realisation of $U_t$.
  \item A causal \emph{safety certificate} recording the repaired action,
    reliability state, uncertainty state, and assumption tags.
\end{enumerate}

None of these objects require SOC, voltage, or Coulomb counting.  Battery
semantics live in the battery instantiation, while the reusable universal
claim attaches to the typed kernel shared across domains.

\section{What Changes by Domain}

Porting ORIUS to a new domain requires replacing four domain-specific
components:

\begin{description}
  \item[Plant dynamics.] The battery uses a linear SOC integrator;
    a robotic arm requires nonlinear rigid-body dynamics.  The pipeline
    is agnostic---it consumes the predicted state set, not the equations.
  \item[State definition.] SOC is a scalar in $[0,1]$.  An autonomous
    vehicle has a six-degree-of-freedom pose.  The conformal region
    $C_t^{\text{RAC}}$ generalises to a set in $\mathbb{R}^n$.
  \item[Constraint structure.] Battery constraints are box constraints
    on SOC and power.  Other domains may have polytopic, conic, or
    nonlinear constraints.  The safe-action filter requires only that
    projection onto the feasible set is computationally tractable.
  \item[Fault taxonomy.] Dropout, delay, drift, and stale-sensor faults
    are battery-specific.  Each domain has its own fault families---GPS
    multipath for vehicles, encoder drift for robots---but the OQE
    scoring architecture applies unchanged.
\end{description}

\section{What Remains Invariant Across Domains}

Four structural invariants transfer without modification:

\begin{enumerate}
  \item \textbf{OQE architecture.}  The feature vector changes; the
    weighted scoring function and calibration procedure do not.
  \item \textbf{RAC-Cert inflation.}  The conformal prediction interval
    is domain-agnostic.  Reliability-adaptive width scaling
    $\hat{q}_\alpha / w_t$ applies to any residual distribution.
  \item \textbf{Repair projection.}  The safe-action filter projects
    onto the feasible set under worst-case uncertainty.  The projection
    operator changes with the constraint geometry; the principle of
    worst-case projection does not.
  \item \textbf{Certification logic.}  The certificate records
    $(w_t, C_t^{\text{RAC}}, a_t^{\text{safe}}, \text{margin})$
    regardless of what the symbols denote physically.
\end{enumerate}

\section{How to Port ORIUS to a New Domain}

A practitioner deploys ORIUS on a new plant by implementing a
\texttt{DomainAdapter}---the analogue of \texttt{BatteryDomainAdapter}.
The adapter provides:

\begin{enumerate}
  \item A telemetry parser that produces $z_t$.
  \item An OQE feature extractor for the domain's fault families.
  \item A plant model (or learned surrogate) for conformal residual
    calibration.
  \item A constraint specification and projection oracle.
\end{enumerate}

The ORIUS core---OQE scoring, RAC inflation, certificate issuance---remains
a shared library, tested once and reused rather than reimplemented per domain.

\section{Cross-Domain Empirical Validation}
\label{sec:cross-domain-eval}

To move beyond architecture-only claims, we validated the ORIUS universal
pipeline on a single canonical replay-backed evidence surface.  The
canonical validator now combines: (i) the locked battery reference
aggregate from publication artifacts; and (ii) a shared universal replay
harness over autonomous-vehicle, industrial, healthcare, navigation, and
aerospace telemetry.  Each row uses the same shared runtime path, but the
claim tier is governed by evidence quality rather than by the existence of
an adapter alone.

\subsection{Datasets}

\begin{itemize}
  \item \textbf{Energy Management} uses the locked DE ENTSO-E dispatch
    surface (12{,}163 training observations, 2{,}537 test windows)
    under degraded-telemetry fault injection matching the universal
    fault schedule.  The energy management instantiation is developed
    in full in Chapters~\ref{ch:forecasting}--\ref{ch:no-free-safety};
    this chapter presents it as the reference witness rather than as one
    peer among equivalent domains.
  \item \textbf{Autonomous Vehicles} use locked longitudinal kinematic
    telemetry with speed-limit constraints.  The training surface
    (85 train / 6 calibration / 19 test) is locked and verified.
  \item \textbf{Industrial Process Control} uses locked CCPP process
    telemetry (6{,}680 train / 477 calibration / 1{,}433 test) with
    temperature and power safety constraints.
  \item \textbf{Medical Monitoring -- ICU Vitals} uses locked PhysioNet
    BIDMC patient telemetry (319 train / 22 calibration / 71 test) with
    heart-rate and SpO\textsubscript{2} constraints.
  \item \textbf{Aerospace} uses locked flight-envelope telemetry
    (3{,}483 train / 248 calibration / 748 test) with airspeed and
    bank-angle constraints.
  \item \textbf{Navigation} uses a closed-loop 2D simulation surface
    retained as portability evidence; it does not carry a trained
    forecasting model and appears in the multi-domain comparison table only.
\end{itemize}

\subsection{Protocol}

All six domains are evaluated under a shared fault schedule: 15\%
packet dropout, 8\% spike-to-unsafe-value corruption, and 10\% stale
readings.  For each domain, $N = 2000$ telemetry steps are replayed
through the DC3S pipeline; the candidate action and the repaired action
are evaluated under the same row-wise truth surface.  We report:
baseline TSVR, post-shield TSVR, repair rate, mean reliability
$\bar{w}_t$, and latency.  Promotion from \emph{proof candidate} to
\emph{proof validated} requires six conditions simultaneously: locked
non-synthetic telemetry, a verified training surface, a clean
software-in-loop (SIL) runtime pass, a nontrivial baseline gap, material
TSVR reduction after repair, and stable behavior across seeds.

TSVR is a structural artefact, not a conformal-coverage shortfall.
Navigation provides simulation-backed portability evidence, and aerospace
remains experimental pending promotion-gate completion.  Energy management
remains the reference row.

\subsection{Rule-Based Baseline Comparison}

To characterise where DC3S adds value, each domain was additionally
evaluated with a domain-appropriate rule-based controller operating
\emph{without} the DC3S shield: a proportional speed governor~(AV),
a PI setpoint controller targeting the 465~MW midpoint~(industrial),
a clinical threshold protocol based on heart-rate ranges~(healthcare),
a proportional bank limiter with 80\% damping~(aerospace), and a
center-guidance controller returning to the safe corridor
midpoint~(navigation).  The results appear in the ``Rule-Based~(\%)''
column of Table~\ref{tab:all_domain_comparison}.

Two distinct findings emerge.  For AV~(48.8\%) and healthcare~(7.5\%),
the rule-based controllers still produce non-trivial TSVR under fault
injection because the heuristic thresholds do not adapt to degraded
telemetry; DC3S is therefore \emph{necessary} in those domains to achieve
the sub-$\alpha$ violation guarantee.  For aerospace, industrial, and
navigation, the rule-based controllers achieve 0\% TSVR by always
outputting maximally conservative actions (clipping to constraint
midpoints or applying heavy damping), but this safety comes at the cost
of sacrificing all control performance.  DC3S achieves equivalent safety
in these domains while permitting the learned, performance-optimising
candidate controller to operate near the Pareto frontier---the
OQE-aware inflation adjusts constraint margins dynamically rather than
anchoring them at the most conservative feasible point.  In both cases,
DC3S is either necessary for safety (AV, healthcare) or necessary for
the combination of safety \emph{and} performance (aerospace, industrial,
navigation).

\subsection{Implications}

These results support three claims:

\begin{enumerate}
  \item \textbf{Portability without re-engineering.} Each new domain required
    only a typed domain instantiation; the shared kernel in
    \path{src/orius/universal_theory/} was not modified.  All adapters share
    the same five-stage runtime, the same reliability-to-uncertainty logic,
    and the same certificate surface.
  \item \textbf{Tiered evidence, not equal guarantees.} The defended claim
    surface is battery reference plus industrial/healthcare proof-validation.
    AV remains proof-candidate, navigation remains shadow-synthetic, and
    aerospace remains experimental.  The common result is a reusable
    degraded-observation repair contract under one protocol, not six domains
    with equivalent proof depth.
  \item \textbf{Overhead is negligible.} Sub-millisecond pipeline latency
    (P95~$<\!0.1$~ms across all domains) makes ORIUS suitable for
    fast-control-loop applications (AV at 1~Hz, industrial at hourly cadence,
    healthcare at continuous monitoring, aerospace at avionics cadence).
\end{enumerate}

\textbf{Physical context.}
The energy management domain models a grid-scale battery dispatch problem
where a storage controller must keep state-of-charge~(SOC) within safe

\subsection{Aerospace (Flight Envelope)}
\label{sec:profile-aerospace}

\textbf{Physical context.}
The aerospace domain monitors a fixed-wing aircraft flight envelope.
The target state includes airspeed ($V_t$, kt) and bank angle
($\phi_t$, degrees).  Safety constraints enforce $V_t \in [60, 350]$~kt
(stall-to-\textit{V}$_{\text{NE}}$ envelope) and
$|\phi_t| \leq 20^\circ$ (operational maneuvering bank limit).  Faults model
air-data computer dropouts, pitot-static spike events, and stale ADIRU
readings.

\textbf{Adapter parameters.}
Constraints: $\mathtt{v\_min\_kt} = 60$, $\mathtt{v\_max\_kt} = 350$,
$\mathtt{bank\_limit\_deg} = 20.0^\circ$;
fallback action: hold throttle, wings-level command;
repair operator: throttle clamping and bank-angle projection to
$[-20^\circ, +20^\circ]$.

\textbf{DC3S result.}
Baseline TSVR~26.65\% $\rightarrow$ 0.0\% post-shield (77.3\% repair rate,
mean $\bar{w}_t = 0.782$, $N = 2000$).

\textbf{Violation predicate scope.}
The violation predicate $v(\cdot)$ evaluates the \emph{commanded bank angle}
only; airspeed is a plant state that cannot be instantaneously modified by a
single DC3S action, so it is excluded from the per-step repair gate consistent
with the single-step DC3S action model.  Airspeed constraints are enforced
through the feasible throttle set $\mathcal{A}_t$, not the per-step violation
check.

\section{Chapter Summary}

The DC3S pipeline operates on domain-abstract objects---observations,
reliability scores, conformal regions, feasible sets, and projected
actions---and is therefore structurally portable across CPS domains.
What changes across domains is the plant-side adapter; what remains
invariant is the safety architecture.

Empirical validation across the tiered six-domain package confirms that
DC3S reduces true-state violations on the strongest replay rows while
preserving sub-millisecond overhead.  Per-domain figures
(\S\ref{sec:per-domain-profiles}) show the common forecasting and runtime
artifact template: locked test splits where available, trained GBM
surfaces for the trained rows, 90\% conformal prediction bands, and OQE
reliability traces under fault injection.  The training audit
(Table~\ref{tab:all-domain-training-metrics}) and software-in-loop table
(Table~\ref{tab:domain-sil-summary}) confirm that each domain's
forecasting and runtime surfaces are reproducible and execution-verified
at its assigned evidence tier.  All six domains share the same
evaluation protocol, artifact types, and evidence tier structure.  The
theorem-to-code-to-artifact lineage
established in the preceding chapters for energy management
(Chapters~\ref{ch:forecasting}--\ref{ch:no-free-safety}) applies equally
to each domain through the shared adapter contract.  Navigation provides
additional simulation-backed portability evidence.
